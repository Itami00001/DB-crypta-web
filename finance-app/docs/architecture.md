# Архитектура приложения: слои и компоненты

## Обзор архитектуры

Приложение построено по принципу многослойной архитектуры (Layered Architecture) с четким разделением ответственности между компонентами. Это обеспечивает модульность, тестируемость и масштабируемость.

```
┌─────────────────────────────────────────────────────────────┐
│                    Presentation Layer                       │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐  │
│  │   Routes    │  │ Middleware  │  │     Swagger         │  │
│  │   (API)     │  │ (Auth, Val) │  │   Documentation    │  │
│  └─────────────┘  └─────────────┘  └─────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
                              │
┌─────────────────────────────────────────────────────────────┐
│                    Business Logic Layer                     │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐  │
│  │ Controllers │  │   Services  │  │   Transaction      │  │
│  │  (Thin)     │  │  (Business) │  │   Management       │  │
│  └─────────────┘  └─────────────┘  └─────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
                              │
┌─────────────────────────────────────────────────────────────┐
│                     Data Access Layer                       │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐  │
│  │   Models    │  │ Associations│  │   Database          │  │
│  │  (Sequelize)│  │  (Relations)│  │  (PostgreSQL)       │  │
│  └─────────────┘  └─────────────┘  └─────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

## Слой 1: Presentation Layer (Уровень представления)

### 1.1. Routes (Маршруты)
- **Расположение:** `app/routes/*.js`
- **Ответственность:** Определение REST API эндпоинтов
- **Компоненты:**
  - `user.routes.js` — управление пользователями
  - `transaction.routes.js` — финансовые транзакции
  - `crypto-wallet.routes.js` — крипто-кошельки
  - `news-post.routes.js` — новостные посты
  - `comment.routes.js` — комментарии
  - `like.routes.js` — лайки
  - `user-prediction.routes.js` — прогнозы пользователей
  - `chart-point.routes.js` — точки на графиках

### 1.2. Middleware (Промежуточное ПО)
- **Расположение:** `app/middleware/*.js`
- **Компоненты:**
  - `auth.jwt.js` — JWT аутентификация
  - `validation.js` — валидация входных данных

### 1.3. Swagger Documentation
- **Расположение:** `server.js` (конфигурация)
- **Ответственность:** Автоматическая генерация API документации
- **Функции:** Интерактивные схемы, примеры, описание ошибок

## Слой 2: Business Logic Layer (Уровень бизнес-логики)

### 2.1. Controllers (Контроллеры)
- **Расположение:** `app/controllers/*.js`
- **Ответственность:** Обработка HTTP запросов, вызов сервисов
- **Принцип:** "Thin controllers" — минимальная логика, делегирование сервисам
- **Пример:** `transaction.controller.js` обрабатывает `transferBetweenUsers`, но делегирует бизнес-логику в `TransactionService`

### 2.2. Services (Сервисы)
- **Расположение:** `app/services/*.js`
- **Ответственность:** Основная бизнес-логика приложения
- **Компоненты:**
  - `user.service.js` — управление пользователями, балансами, аутентификация
  - `transaction.service.js` — финансовые операции, транзакции
- **Функции:**
  - Валидация бизнес-правил
  - Управление транзакциями БД
  - Обработка ошибок бизнес-логики
  - Инкапсуляция сложных операций

### 2.3. Transaction Management
- **Уровень изоляции:** `SERIALIZABLE` для финансовых операций
- **Блокировки:** Row-level locks (`FOR UPDATE`)
- **Обработка конфликтов:** Retry логика для serialization failures

## Слой 3: Data Access Layer (Уровень доступа к данным)

### 3.1. Models (Модели)
- **Расположение:** `app/models/*.js`
- **Ответственность:** Определение схемы данных, валидации, ассоциации
- **Особенности:**
  - UUID первичные ключи
  - Валидации на уровне модели (`min: 0` для балансов)
  - Индексы для оптимизации запросов
- **Компоненты:**
  - `user.model.js` — пользователи
  - `transaction.model.js` — транзакции
  - `crypto-wallet.model.js` — кошельки
  - И другие модели сущностей

### 3.2. Associations (Ассоциации)
- **Расположение:** `app/models/associations.js`
- **Ответственность:** Определение отношений между моделями
- **Типы отношений:**
  - One-to-Many (User → CryptoWallet)
  - Many-to-Many (через связующие таблицы)
  - Self-referencing (Comments → parentComment)

### 3.3. Database (База данных)
- **СУБД:** PostgreSQL
- **Особенности:**
  - UUID с `gen_random_uuid()`
  - Индексы для производительности
  - Расширение `pgcrypto` для UUID
- **Схемы:**
  - `finance_app_db` — оригинальная (integer)
  - `finance_app_db_uuid` — новая (UUID)

## Взаимодействие между слоями

### Flow запроса (пример: перевод средств)
1. **Routes:** `POST /api/transactions/transfer`
2. **Middleware:** JWT аутентификация + валидация
3. **Controller:** `transaction.controller.js` → `transferBetweenUsers`
4. **Service:** `TransactionService.transferFunds()` с SERIALIZABLE транзакцией
5. **Models:** `User`, `CryptoWallet`, `Transaction` с блокировками
6. **Database:** PostgreSQL с ACID гарантиями

### Обработка ошибок
- **Presentation:** HTTP статусы (400, 401, 403, 404, 409, 500)
- **Business:** Бросают исключения с описанием
- **Data:** Sequelize ошибки валидации

## Преимущества архитектуры

### 1. Разделение ответственности
- Каждый слой имеет четкую зону ответственности
- Легко понимать, где находится определенная логика

### 2. Тестируемость
- Сервисы можно тестировать изолированно
- Controllers можно мокать через сервисы

### 3. Масштабируемость
- Новые бизнес-правила добавляются в сервисы
- Новые API эндпоинты в контроллеры
- Новые сущности в модели

### 4. Поддерживаемость
- Изменения в одном слое минимально влияют на другие
- Четкая структура помогает новым разработчикам

## Безопасность

### 1. Аутентификация и авторизация
- JWT токены в middleware
- Проверка прав доступа в сервисах

### 2. Валидация данных
- Схемы валидации в middleware
- Бизнес-валидация в сервисах
- Constraints на уровне БД

### 3. Транзакционная безопасность
- SERIALIZABLE изоляция для финансовых операций
- Row-level блокировки предотвращают race conditions
- ACID свойства обеспечивают целостность данных

## Будущие улучшения

### 1. Event-driven архитектура
- События для асинхронной обработки
- Event sourcing для аудита

### 2. Microservices
- Выделение отдельных сервисов
- API Gateway для маршрутизации

### 3. Caching
- Redis для кэширования балансов
- Кэширование запросов

### 4. Monitoring
- Логирование операций
- Метрики производительности

Эта архитектура обеспечивает надежную основу для финансового приложения с возможностью дальнейшего развития и масштабирования.
